# Docker testbed

Local network of N qBitcoin nodes, with `node-0` as the bootnode and `node-1..node-N` as peers that auto-discover its multiaddr at startup. The compose file is generated, not hand-edited — re-run the generator whenever you change topology.

---

## 1. Quick start

```sh
./scripts/gen-compose.sh 8           # node-0 (bootnode) + 8 mining peers
./scripts/gen-compose.sh 16 nomine   # 16 peers, only the bootnode mines
docker compose up --build
```

Stop and wipe state:

```sh
docker compose down -v               # the -v drops volumes — fresh chain next start
```

---

## 2. What's inside

| File | Purpose |
|---|---|
| `Dockerfile` | Multi-stage: build on `golang:alpine`, ship on `alpine:3.20` under `tini` with `qbitcoind` + `qbitcoin-cli` + `curl`. |
| `docker/entrypoint.sh` | Branches on `NODE_ROLE` to start as bootnode / peer-mine / peer-only. |
| `docker-compose.yml` | Generated by `scripts/gen-compose.sh`. Each node is its own service (not `--scale`-replica) so `lazydocker` and `docker compose logs node-N` show per-node output. |
| `scripts/gen-compose.sh` | Regenerates `docker-compose.yml` for a given peer count. Optional second arg `nomine` disables mining on peers. |

---

## 3. Roles (`NODE_ROLE`)

`docker/entrypoint.sh` switches on `NODE_ROLE`:

| Role | What happens |
|---|---|
| `bootnode` | Tees logs through `tee /tmp/node.log`, scrapes the `p2p listen: …/p2p/<peerid>` line, writes `/dns4/<host>/tcp/8333/p2p/<peerid>` to a shared `bootstrap_addr` volume. Starts with `--bootnodes none`. Creates a wallet + mines. |
| `peer-mine` | Waits on `/bootstrap/bootnode.addr`, then starts with `--bootnodes <addr>`. Owns its wallet + mines. |
| `peer` | Stateless validator. Waits on the bootstrap addr, has no wallet, doesn't mine. |

Wallets in the testbed are unencrypted + autoload so containers restart without passphrase prompts.

Each container binds RPC to `0.0.0.0` (entrypoint passes `-rpcbind 0.0.0.0`) so the host port mapping reaches the daemon's listener. Access is still gated by the per-container `.cookie` under `/data`, so the exposed port is safe: only someone with read access to that container's volume can hit the RPC. For in-container CLI calls (entrypoint scripts + `docker compose exec`) the cookie is on the same volume, so `qbitcoin-cli -datadir /data …` "just works".

---

## 4. Port mapping

The generator maps each node's RPC to a unique host port:

| Container | RPC port (container) | RPC port (host) |
|---|---|---|
| `node-0` (bootnode) | 8334 | 8334 |
| `node-1` | 8334 | 8335 |
| `node-2` | 8334 | 8336 |
| … | … | … |
| `node-N` | 8334 | 8334 + N |

So `qbitcoin-cli -rpc http://localhost:8336 getblockcount` queries node-2. Because the cookie lives inside each container's volume (not on the host), from-the-host calls need either `-rpcuser`/`-rpcpassword` (set via `-rpcuser`/`-rpcpassword` env on the container) or a cookie copied out of the volume. The natural pattern is `docker compose exec node-N qbitcoin-cli -datadir /data …` (shown in §5), which stays inside the container and finds the cookie automatically.

P2P traffic stays inside the compose network — no host port-mapping for libp2p ports.

## 5. Controlling a specific peer

```sh
# Mining is out-of-process — run qbitcoin-miner against a specific peer's RPC:
docker compose exec node-2 qbitcoin-miner -datadir /data -coinbase qbtc1q… -threads 4

# Regtest-style helper if you just need N blocks:
docker compose exec node-2 qbitcoin-cli -datadir /data generatetoaddress 10 qbtc1q…

# Wallet ops:
docker compose exec node-2 qbitcoin-cli -datadir /data getbalance
docker compose exec node-2 qbitcoin-cli -datadir /data send qbtc1q… 100000

# Logs:
docker compose logs -f node-2
```

`-datadir /data` makes the CLI find the RPC cookie; the entrypoint itself uses the same flag under the hood. You can alias `qbitcoin-cli -datadir /data` inside the container if you're running many commands.

`docker compose logs node-N` works because each node is a separate service. `docker compose logs` (no service) tails everything interleaved.

---

## 6. Regenerating the compose file

`scripts/gen-compose.sh <peers> [nomine]`:

- `peers` — number of non-bootnode peers (so total = `peers + 1`).
- `nomine` (optional) — peers run as stateless validators (`NODE_ROLE=peer`) instead of mining.

Always re-run after changing the count, then `docker compose up --build`.

---

## 7. Wipe and restart

```sh
docker compose down -v               # drop all per-node volumes
./scripts/gen-compose.sh 8           # regenerate (optional if topology unchanged)
docker compose up --build            # fresh chain from genesis
```

The `-v` is necessary to clear Pebble state — without it, peers come back up at their previous tip.

---

## 8. Use cases

The testbed is designed for end-to-end flow exercises that single-process tests can't cover:

- Compact-block reconstruction across peers.
- Reorg handling under concurrent mining.
- Mempool eviction on block confirmation across peers.
- BIP-152 short-ID collision recovery.
- Ban-manager threshold crossings under adversarial fixtures.

For unit-level coverage of any subsystem, prefer the in-tree tests under each package.
